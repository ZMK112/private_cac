Docker 模式会隔离主 Claude Code 容器：使用独立网络命名空间、sing-box TUN 透明代理、cac 身份保护，并预装 Claude Code。

如果你在 `cac` 容器里启用了子容器工作流，这部分便利性来自通过 sidecar 代理访问宿主 Docker Engine。主 `cac` 容器仍然隔离，但子容器不会自动继承主容器的 leak-safe 网络路径。

## 快速开始

```bash
cd /path/to/your/project
cac docker setup     # 粘贴你的代理地址
cac docker create    # 如果在仓库里运行，则构建本地镜像
cac docker start     # 启动容器
cac docker enter     # 进入容器 Shell — claude + cac 已就绪
```

执行 `cac docker start` 后，宿主当前目录会挂载到主容器内的 `/workspace`。

## 设置

`cac docker setup` 是一个交互式向导，它会：

1. 询问你的代理地址（支持紧凑格式和分享链接格式）
2. 自动检测本地或远程 Docker 安装
3. 自动检测 macvlan 模式的网络参数（网卡、子网、网关）
4. 将配置写入 `docker/.env`

如果你通过 `bash install.sh --local --yes` 做的是本地安装，这通常指的是 `~/.cac/docker/.env`；如果你直接在源码树中运行，则是 `<repo>/docker/.env`。

### 代理格式

```
ip:port:user:pass          # 紧凑格式 SOCKS5
host:port                  # 使用主机名的紧凑格式 SOCKS5
ss://...                   # Shadowsocks 分享链接
vmess://...                # VMess 分享链接
vless://...                # VLESS 分享链接
trojan://...               # Trojan 分享链接
```

### Docker Desktop 上的本地 SOCKS5

如果你的代理程序直接跑在 macOS 宿主机上，主容器应当通过 `host.docker.internal` 访问它：

```env
PROXY_URI=host.docker.internal:17891
CAC_CHILD_CONTAINER_PROXY_URL=socks5h://host.docker.internal:17891
CAC_CHILD_CONTAINER_NO_PROXY=localhost,127.0.0.1,::1,host.docker.internal
```

`BUILD_PROXY` 是可选的，只影响镜像构建阶段的下载；运行时的 TUN 流量由 `PROXY_URI` 控制，不由 `BUILD_PROXY` 控制。

## 容器内部发生了什么

首次启动时，入口脚本会自动：

<Steps>
  <Step title="启动 sing-box TUN">
    所有容器流量在内核层面透明地通过你的代理路由。应用程序看不到代理环境变量。
  </Step>
  <Step title="检测时区和语言">
    查询出口 IP 以确定时区和语言，然后配置容器。
  </Step>
  <Step title="创建 cac 配置">
    生成唯一身份标识（UUID、主机名、MAC、机器 ID）和 mTLS 证书。
  </Step>
  <Step title="激活配置">
    设置环境变量并写入 `/root/.cac-env` 以在 Shell 会话间持久化。
  </Step>
</Steps>

## 持久化存储

`/root` 和 `/home` 会通过 bind mount 挂载到 Docker 资源目录下的 `data/`。你的 Claude Code 会话、设置和工作文件在容器重启后依然保留。

`/workspace` 会从你执行 `cac docker start` 时所在的宿主目录挂载进容器。如果 Claude Code 需要启动带 bind mount 的子容器，应当在你希望暴露的项目根目录下启动 Docker 模式。

## `cac` 内部启动子容器

在容器里，`docker` 是对真实 Docker CLI 的一层包装：

- 它通过专用的 `docker-proxy` sidecar 访问宿主 Docker Engine
- 会把 `/workspace/...` 的 bind mount 改写为宿主对应路径
- 默认给子容器使用 `bridge` 网络，除非你显式传 `--network`
- 会自动打标签，方便识别和清理 Claude 创建的子容器

需要明确的网络边界：

- `docker pull` 和 `docker build` 走的是宿主 Docker Engine，不是主 `cac` 容器自己的 TUN 路径
- 子容器运行时流量默认也是宿主 Docker Engine 的普通网络，不会自动跟随主容器代理
- 如果你希望 registry 流量走代理，需要单独给 Docker Engine 或 Docker Desktop 配 daemon 代理
- 如果你希望子容器里的应用流量走代理，需要在 `docker/.env` 里设置 `CAC_CHILD_CONTAINER_PROXY_URL`，或者自己传 `-e ALL_PROXY=... -e HTTPS_PROXY=...`
- 如果主代理中断，主容器预期是 fail-closed：`cac docker check` 的 DNS/TCP/HTTP 检查会失败，而不是静默回退到宿主默认出口
- 在本地 SOCKS5 测试里，更关键的 fail-closed 信号是代理停掉后 `HTTP` 和 `EXIT` 失败，且不会暴露新的直连出口 IP

这些 `docker/templates/cherny.*` 文件属于 Docker 模式的源代码资产，应该和代码一起提交，而不是只留在本地工作区。

重要说明：

- 这一模式下，主 `cac` 容器仍然是隔离最强的一层
- 但容器内的 Docker 访问本身不再是强隔离边界，因为 Claude Code 可以请求宿主 Docker Engine 创建其他容器
- 不要假设子容器在项目代理失效时仍然 fail-closed；如果你需要这种行为，必须分别配置 daemon 代理和子容器代理环境变量

## 端口转发

要从容器暴露端口（例如开发服务器）：

```bash
cac docker port 6287
```

这会启动一个 socat/python3 中继，将主机的 `localhost:6287` 转发到容器。

## 诊断

```bash
cac docker check
```

检查 TUN 状态、DNS、TCP/HTTP 连通性、出口 IP、泄露检测和身份伪装。
