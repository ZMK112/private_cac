## 命名规范

Docker 模式统一使用 `boris-` 前缀来替代 `cac`，所以主容器叫 `boris-main`，代理 sidecar 是 `boris-gateway`，设备主机名格式是 `boris-<short-uuid>`，工作区挂载在 `/workspace`。

模板文件 `docker/templates/cherny.*` 中冻结了 `TZ=Asia/Tokyo` 和 `LANG=ja_JP.UTF-8`，并对 `env`、prompt、telemetry` 的内容做了统一，避免将真实环境/设备信息汇报给 Anthropic。

## 构建与启动

```bash
./cac docker create
./cac docker start
```

结果是 `.env` 中固定的 `CAC_DOCKER_IMAGE` 镜像引用会被构建并使用（例如 `ghcr.io/nmhjklnm/cac-docker:v1.4.4-zmk.11`），同时代理镜像也会构建完成，`boris-gateway` 与 `boris-main` 成功运行，`/workspace` 绑到宿主当前目录。

- `cac docker port`
- 上游代理失效后的 fail-closed 行为

## 这轮验证中修掉的根因

1. `docker/entrypoint.sh` 里的 `apply_cherny_identity()` 会在 `set -e` 下返回失败，导致主容器在 profile 创建后反复重启。
2. 本地 `cac docker port` 过去依赖宿主直连容器 bridge IP，这在 Docker Desktop 场景并不稳定。
3. `cac docker check` 过去依赖附着式 exec 路径，长时间网络探测时容易中断。

当前实现对应修复为：

- `apply_cherny_identity()` 使用 `mapfile` 读取全部字段，并显式返回成功
- 本地模式改为 helper container + published localhost port
- `cac docker check` 改为容器内后台执行 `cac-check`，再统一回读输出
