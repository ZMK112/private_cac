Docker mode isolates the main Claude Code container with a separate network namespace, sing-box TUN transparent proxy, cac identity protection, and Claude Code pre-installed.

If you enable child-container workflows from inside `cac`, that convenience comes from talking to the host Docker Engine through a sidecar proxy. The main `cac` container stays isolated; child containers do not automatically inherit the main container's leak-safe network path.

## Quick start

```bash
cd /path/to/your/project
cac docker setup     # paste your proxy address
cac docker create    # build the local image if running from the repo
cac docker start     # start container
cac docker enter     # shell in — claude + cac ready
```

After `cac docker start`, the current host directory is mounted to `/workspace` inside the main container.
The default runtime user remains non-root, but now has passwordless `sudo` for ad hoc package installs and one-off maintenance inside the container.
The base image also preinstalls a small data/dev CLI set: `jq`, `ripgrep`, `fd`, `tree`, `sqlite3`, `zip`, and `unzip`.
Docker mode also publishes an SSH endpoint on `127.0.0.1:2222` by default so the host can connect straight into the main container without extra key setup.

## Setup

`cac docker setup` is an interactive wizard that:

1. Asks for your proxy address (supports compact and share link formats)
2. Auto-detects local vs remote Docker installation
3. Auto-detects network parameters (interface, subnet, gateway) for macvlan mode
4. Writes configuration to `docker/.env`

After a local install via `bash install.sh --local --yes`, that usually means `~/.cac/docker/.env`. When you run directly from the source tree, it means `<repo>/docker/.env`.

### Proxy formats

```
ip:port:user:pass          # compact SOCKS5
host:port                  # compact SOCKS5 with hostname
ss://...                   # Shadowsocks share link
vmess://...                # VMess share link
vless://...                # VLESS share link
trojan://...               # Trojan share link
```

### Local SOCKS5 on Docker Desktop

If your proxy runs on the macOS host itself, point the main container at `host.docker.internal`:

```env
PROXY_URI=host.docker.internal:17891
CAC_CHILD_CONTAINER_PROXY_URL=socks5h://host.docker.internal:17891
CAC_CHILD_CONTAINER_NO_PROXY=localhost,127.0.0.1,::1,host.docker.internal
```

`BUILD_PROXY` is optional and only affects image build-time downloads. Runtime TUN traffic is controlled by `PROXY_URI`, not `BUILD_PROXY`.

## What happens inside the container

On first boot, the entrypoint automatically:

<Steps>
  <Step title="Start sing-box TUN">
    All container traffic is transparently routed through your proxy at the kernel level. No proxy environment variables visible to applications.
  </Step>
  <Step title="Detect timezone and locale">
    Queries the exit IP to determine timezone and language, then configures the container.
  </Step>
  <Step title="Create cac profile">
    Generates a unique identity (UUID, hostname, MAC, machine ID) and mTLS certificate.
  </Step>
  <Step title="Activate profile">
    Sets environment variables and writes to `/root/.cac-env` for persistence across shell sessions.
  </Step>
</Steps>
The bootstrap identity always uses the Asia/Tokyo timezone and `ja_JP.UTF-8` locale that the templates ship with, so Claude Code starts with a consistent `TZ`/`LANG` pair regardless of your host. The generated hostname follows the `boris-<short-uuid>` pattern and the workspace bind mount points to whatever directory you used when running `cac docker start`.

## Persistent storage

`/root` and `/home` are bind-mounted to the Docker resource directory's `data/` folder on the host. Your Claude Code sessions, settings, and work files survive container restarts.

`/workspace` is bind-mounted from the directory where you ran `cac docker start`. If Claude Code needs to start child containers with bind mounts, start Docker mode from the project root you want exposed inside `/workspace`.

## Runtime user

The main shell inside the container runs as the canonical non-root persona user (`cherny` by default). That user has passwordless `sudo`, so commands like `sudo apt-get update` and `sudo apt-get install ...` work without switching the default runtime model back to root.

## SSH access

Docker mode starts `sshd` inside the main container and publishes it to `127.0.0.1:${CAC_HOST_SSH_PORT:-2222}` on the host. Password login is enabled for the runtime user, and the default password is `cherny` unless you override `CAC_SSH_PASSWORD` in `docker/.env`, so the common path is:

```bash
ssh -p 2222 cherny@127.0.0.1
```

The published port stays localhost-only by default; it is not exposed to the rest of the LAN unless you intentionally change the port binding.

Public-key auth is also left enabled for later use, but `cac` does not import any host SSH keys or mount the host `~/.ssh`. If you want key-based login later, add your own key manually inside the persisted container home, for example `/home/cherny/.ssh/authorized_keys`.

## Child containers from inside `cac`

Inside the container, `docker` is a wrapper around the real Docker CLI:

- It talks to the host Docker Engine through a dedicated `docker-proxy` sidecar
- It rewrites bind mounts from `/workspace/...` to the matching host path
- It defaults child containers to `bridge` unless you pass `--network` yourself
- It adds labels so Claude-started containers are easy to identify and clean up

Important network boundary details:

- `docker pull` and `docker build` run through the host Docker Engine, not through the main `cac` container's TUN path
- Running child containers also use the host Docker Engine's normal networking unless you explicitly set proxy environment variables inside those child containers
- If you want registry traffic to use a proxy, configure the Docker Engine or Docker Desktop daemon proxy separately
- If you want a child container's application traffic to use a proxy, set `CAC_CHILD_CONTAINER_PROXY_URL` in `docker/.env` or pass `-e ALL_PROXY=... -e HTTPS_PROXY=...` yourself
- If the main proxy goes down, the main container is expected to fail closed: `cac docker check` will fail on DNS/TCP/HTTP instead of silently falling back to the host's default route
- In local SOCKS5 tests, the strongest fail-closed signal is that `HTTP` and `EXIT` fail after the proxy stops; no direct exit IP is exposed
## Defense-in-depth and canonical identity

Docker mode mirrors the three-layer architecture that projects like CC Gateway and the ByteDance audit highlight: env-vars, host-level firewalling, and identity rewriting.

- **Identity rewrite.** Every profile generated inside `/usr/local/share/cherny` anonymizes `device_id`, email, `user_id`, `session_alias`, `hostname`, and even TLS client certificates. The default names are anchored to the `boris-...` template, so nothing mentions `cac` on the public telemetry fingerprint.
- **Environment normalization.** The `cherny.env.json` snapshot pins `TZ=Asia/Tokyo`, `LANG=ja_JP.UTF-8`, and JavaScript `process.env` values (platform, arch, runtimes) that Claude Code would otherwise leak in its `env` block.
- **Prompt and telemetry sanitization.** As noted in the bytedance leak report, Claude Code injects `<env>` metadata into every system prompt and ships it to Datadog / GrowthBook. Our copy of the `cherny.prompt.json` template rewrites those fields to match the canonical profile, and the `cherny.telemetry.json` payload masks constrainedMemory/RSS so vendor-side heuristics see only the allowed values.

These `docker/templates/cherny.*` files are part of the source tree and should be committed like code, not treated as local-only scratch assets.

Coupled with the docker wrapper that injects `CAC_CHILD_CONTAINER_PROXY_URL` and `NO_PROXY` into every Claude-launched container, the main TUN tunnel replicates the CC Gateway defense-in-depth idea: if a proxy dies, `cac docker check` starts failing on DNS/TCP/HTTP instead of silently switching back to the host network.

## Operational checklist

Repeatable validation steps live in the [Docker mode ops log](/guides/docker-mode-ops). The checklist records:

1. Building and synchronizing the pinned `CAC_DOCKER_IMAGE` reference from `.env` with Boris-derived templates. By default, Docker mode now writes a release-tagged image such as `ghcr.io/nmhjklnm/cac-docker:v1.4.4-zmk.11` instead of using `latest`.
2. Starting `boris-main` and the `boris-gateway` sidecar so the workspace mapping and Docker proxy stay consistent.
3. Running `cac docker check` to ensure `tun0` is up, DNS/TCP/HTTP probes behave as expected, and the spoofed hostname is active.

The current ops log also records the validation failures that were fixed during the latest Docker hardening pass: the entrypoint restart bug, the local port-forward implementation, the `cac-check` execution path, and the validation-harness namespace/subnet collisions.

Important:

- The main `cac` container still gets the strongest protection in this mode
- Docker access inside `cac` is no longer a strong isolation boundary, because Claude Code can ask the host Docker Engine to create other containers
- Do not assume child containers are fail-closed when your project proxy is down; configure the Docker daemon proxy and child-container proxy envs explicitly if you need that behavior

## Port forwarding

To expose a port from the container (e.g., a dev server):

```bash
cac docker port 6287
```

In local mode, this starts a small helper container on the compose network and publishes `127.0.0.1:6287` back to the host. In remote/macvlan mode, the command still uses a host-side relay path.

## Diagnostics

```bash
cac docker check
```

Checks TUN status, DNS, TCP/HTTP connectivity, exit IP, leak detection, non-China exit geolocation, SSH availability, and identity spoofing.
