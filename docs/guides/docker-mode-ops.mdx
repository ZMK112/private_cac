## Naming template

Docker mode now uses the `boris-` namespace so nothing in the telemetry stream mentions `cac`. The current pattern is:

- **Main container:** `boris-main`
- **Docker proxy sidecar:** `boris-gateway`
- **Hostnames produced by `docker/templates/cherny.identity.json`:** `boris-<short-uuid>`
- **Workspace mount:** `/workspace` inside the container binds to whatever directory you ran `cac docker start` from on the host.

The canonical identity templates also freeze `TZ=Asia/Tokyo`, `LANG=ja_JP.UTF-8`, and the `env`/`prompt`/`telemetry` blocks that Claude Code would otherwise leak. This mirrors the CC Gateway defense (env vars + Clash rules + gateway rewrite) highlighted by the ByteDance audit of Claude Code telemetry.

## Root causes fixed during validation

1. `apply_cherny_identity()` in `docker/entrypoint.sh` returned a failure status under `set -e`, which caused the main container to restart immediately after profile creation.
2. Local `cac docker port` forwarding relied on host access to the container bridge IP, which is not reliable on Docker Desktop.
3. `cac docker check` was vulnerable to attached exec instability for long-running network probes.

The current implementation fixes these by:

- using `mapfile` plus explicit success return in `apply_cherny_identity()`
- using a helper container to publish localhost ports in local mode
- running `cac-check` asynchronously inside the container and then collecting its output

## Validation harness lessons

Two non-product issues also surfaced while building the repeatable validation flow:

1. Temporary repo copies collided on the default Compose project name because the compose files live under `docker/`.
2. Validation stacks could collide on the Docker bridge subnet.

The current validation script handles both by forcing a unique `COMPOSE_PROJECT_NAME` and probing for a free control subnet before starting the stack.

## Diagnostics and telemetry notes

- The ByteDance document highlights three telemetry channels (Datadog, BigQuery, GrowthBook), 40+ environment dimensions, browser-supplied attribution headers, and the native `cch` attestation token that cannot be forged. Docker mode treats those facts as requirements: the `cherny` templates overwrite the `env`, sanitizes the `prompt` block, and strips fields like `baseUrl`/`gateway` so you never expose the proxy provider.
- Child containers launched through the `docker` wrapper respect `CAC_CHILD_CONTAINER_PROXY_URL` and `CAC_CHILD_CONTAINER_NETWORK_MODE` so their traffic is injected through the same proxy/Hound arrangement, keeping the fail-closed posture intact.
